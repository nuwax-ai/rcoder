## 项目结构与模块划分

本项目采用 Rust Workspace 架构，将功能划分为多个独立的 crate，以实现高内聚、低耦合的设计。各 crate 的职责边界清晰，便于维护和扩展。

- `crates/rcoder`：主应用 crate，负责 HTTP API 路由、会话管理、配置加载、代理启动等核心业务逻辑。
- `crates/pingora-proxy`：封装 Cloudflare Pingora 反向代理，提供高性能的端口路由和后端服务代理功能。
- `crates/claude-code-agent` 和 `crates/codex-acp-agent`：分别实现与 Claude Code 和 OpenAI Codex AI 代理的集成，通过 ACP (Agent Client Protocol) 协议进行通信。
- `crates/acp_adapter`：提供 ACP 协议的适配层，统一处理与不同 AI 代理的交互。
- `crates/nuwax_parser`：项目文件解析器，负责解析和处理项目源代码。
- `crates/shared_types`：共享的数据类型定义，供其他 crate 依赖使用，避免重复定义。

这种模块化设计使得每个 crate 可以独立开发、测试和部署，同时通过清晰的接口进行通信，降低了系统的复杂性。

## 本地开发环境搭建

### 依赖安装

1. **Rust 工具链**：确保安装了 Rust 1.75+ 版本（推荐使用 2024 Edition）。
   ```bash
   rustup install stable
   rustup default stable
   ```

2. **可选依赖**：
   - **Claude Code CLI**：用于 Claude 代理。
     ```bash
     npm install -g @anthropic-ai/claude-code
     # 或
     pip install claude-dev
     ```
   - **OpenAI Codex**：用于 Codex 代理，参考 [Codex 文档](https://github.com/openai/codex)。

3. **环境变量**：配置必要的 API 密钥。
   ```bash
   export ANTHROPIC_API_KEY="your-api-key"
   export ANTHROPIC_MODEL="claude-3-sonnet-20240229"
   ```

### 代码格式化与静态检查

使用 `rustfmt` 进行代码格式化，确保代码风格统一。
```bash
cargo fmt
```

使用 `clippy` 进行静态代码检查，发现潜在的错误和不良实践。
```bash
cargo clippy
# 严格模式，将警告视为错误
cargo clippy -- -D warnings
```

### 测试运行方法

运行所有测试用例。
```bash
cargo test
```

运行特定包的测试。
```bash
cargo test --package rcoder
```

运行集成测试。
```bash
cargo test --test integration
```

## 测试策略

### 单元测试

单元测试位于各 crate 的 `src` 目录下，使用 `#[cfg(test)]` 模块进行组织。每个模块应包含对其公共接口的测试，确保功能的正确性。

### 集成测试

集成测试位于 `tests` 目录下，模拟真实环境下的组件交互。重点测试 HTTP API 接口、代理服务和 AI 代理的集成。

### 覆盖率要求

项目要求单元测试和集成测试的代码覆盖率不低于 80%。使用 `tarpaulin` 工具生成覆盖率报告。
```bash
cargo tarpaulin --out Html
```

## CI/CD流程与代码提交规范

### CI/CD流程

CI/CD 流程由 GitHub Actions 驱动，包含以下步骤：
1. **代码格式化检查**：确保所有代码符合 `rustfmt` 格式。
2. **静态检查**：运行 `clippy` 检查代码质量。
3. **单元测试**：执行所有单元测试。
4. **集成测试**：执行所有集成测试。
5. **覆盖率检查**：生成覆盖率报告并检查是否达标。
6. **Docker 镜像构建**：成功后构建 Docker 镜像并推送到仓库。

### 代码提交规范

1. **分支命名**：使用 `feature/`、`bugfix/`、`hotfix/` 等前缀。
2. **提交信息**：遵循 [Conventional Commits](https://www.conventionalcommits.org/) 规范。
3. **Pull Request**：必须通过 CI 检查，且至少有一名团队成员审查通过后才能合并。

## 调试技巧

### 启用详细日志

通过设置 `RUST_LOG` 环境变量来控制日志级别。
```bash
RUST_LOG=debug cargo run --bin rcoder
```

日志将输出到控制台和 `logs` 目录下的按天滚动的 JSON 文件中，便于后续分析。

### 调试异步代码

使用 `tokio-console` 工具进行异步代码的调试。
```bash
cargo install tokio-console
tokio-console
```

在代码中使用 `tracing` 宏添加跟踪点，以便在 `tokio-console` 中查看任务的执行情况。

### 断点调试

使用 `gdb` 或 `lldb` 进行断点调试。由于项目使用异步运行时，建议在关键的 `async` 函数入口处设置断点，并结合 `tokio-console` 观察任务调度。

## 新功能开发流程与代码审查标准

### 新功能开发流程

1. **需求分析**：明确功能需求和设计目标。
2. **设计评审**：在团队内讨论设计方案，确保符合现有架构。
3. **分支创建**：从 `main` 分支创建新的功能分支。
4. **编码实现**：遵循现有编码风格和设计模式进行开发。
5. **单元测试**：为新功能编写单元测试。
6. **集成测试**：确保新功能与现有系统集成无误。
7. **提交 PR**：创建 Pull Request，等待审查。

### 代码审查标准

1. **代码风格**：符合 `rustfmt` 格式，使用 `clippy` 检查无警告。
2. **功能正确性**：代码逻辑正确，边界条件处理得当。
3. **性能考虑**：避免不必要的性能开销，如过度的内存分配。
4. **可读性**：代码清晰易懂，关键逻辑有适当注释。
5. **测试覆盖**：新功能有充分的单元测试和集成测试。
6. **文档更新**：API 变更需更新相关文档。
