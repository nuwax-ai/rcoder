## 简介
RCoder 项目采用 Conventional Commits 规范来标准化 Git 提交信息。该规范不仅提高了代码历史的可读性，还为自动生成 CHANGELOG 和语义化版本控制提供了基础。本指南详细说明了项目所采用的提交规范，包括提交类型、格式要求以及在多代理模块协同开发时的最佳实践。

## Conventional Commits 规范
Conventional Commits 是一种为提交信息添加结构化约定的规范。它通过标准化的格式帮助开发者快速理解每次提交的目的和影响范围。RCoder 项目严格遵循此规范，确保代码库的历史记录清晰、一致且可自动化处理。

该规范的核心价值在于：
- 提高团队协作效率
- 支持自动化工具链
- 便于生成版本发布说明
- 简化代码审查过程

## 提交类型语义
RCoder 项目采用以下提交类型，每种类型都有明确的语义含义和适用场景：

### feat
表示新增功能。当实现新的用户可见的功能或特性时使用此类型。这包括新 API、新界面元素或新业务逻辑。

**适用场景**：添加聊天消息的多媒体支持、实现新的代理类型集成。

### fix
表示修复缺陷。当修正代码中的错误或问题时使用此类型。这包括逻辑错误、边界条件处理不当等问题。

**适用场景**：修复会话管理中的内存泄漏、修正代理连接超时问题。

### docs
表示文档变更。当修改项目文档时使用此类型，包括 README、API 文档、注释等。

**适用场景**：更新安装指南、完善 API 接口说明、修改代码注释。

### style
表示代码格式调整。当进行不影响代码运行的格式化修改时使用此类型，如缩进调整、空格修改等。

**适用场景**：统一代码缩进风格、调整代码块间距。

### refactor
表示代码重构。当重构代码以提高可读性或性能，但不改变外部行为时使用此类型。

**适用场景**：优化函数结构、改进模块组织、提升代码复用性。

### test
表示测试相关变更。当添加、修改或删除测试代码时使用此类型。

**适用场景**：增加单元测试覆盖率、修复测试用例、更新测试配置。

### chore
表示维护性任务。当进行构建过程、依赖管理或其他辅助性工作时使用此类型。

**适用场景**：更新依赖库版本、调整构建脚本、修改 CI/CD 配置。

## 提交格式要求
RCoder 项目遵循标准的 Conventional Commits 格式，具体要求如下：

### 基本格式
```
<类型>(<作用域>): <简要描述>
```

- **类型**：必须是预定义的提交类型之一（如 feat、fix 等）
- **作用域**：可选，用于指定变更影响的模块或组件
- **简要描述**：简洁明了地描述变更内容，使用现在时态

### 可选正文
当需要提供更详细的解释时，可以在简要描述后添加正文。正文应说明"为什么"要进行此变更，以及变更的上下文。

### 脚注
用于引用相关问题、破坏性变更等信息。常见的脚注包括：
- `Closes #123`：关联问题编号
- `BREAKING CHANGE:`：标记破坏性变更

## 自动生成CHANGELOG与版本管理
遵循 Conventional Commits 规范对自动生成 CHANGELOG 和版本号管理至关重要：

### CHANGELOG 生成
通过分析提交历史中的类型信息，可以自动分类和组织变更内容：
- `feat` 类型的提交归入"新功能"部分
- `fix` 类型的提交归入"问题修复"部分
- `BREAKING CHANGE` 脚注的提交单独列出

### 版本号管理
结合语义化版本控制（SemVer），提交类型直接影响版本号的递增：
- `feat` 提交触发次版本号（minor）递增
- `fix` 提交触发布丁版本号（patch）递增
- 包含 `BREAKING CHANGE` 的提交触发主版本号（major）递增

这种自动化机制确保了版本发布的透明性和可预测性。

## 符合规范的提交示例
以下是符合 RCoder 项目规范的提交信息示例：

```
feat(chat): 添加多媒体消息支持
为聊天接口增加文件上传功能，支持图片和文档类型。
Closes #123
```

```
fix(session): 修复会话超时问题
修正会话管理器在长时间运行后出现的超时异常。
Closes #156
```

```
refactor(agent): 优化代理状态管理
重构代理状态管理模块，提高代码可读性和维护性。
```

```
docs: 更新安装指南
完善项目安装步骤说明，增加环境配置细节。
```

这些示例展示了如何正确使用提交类型、作用域和描述，以及如何关联问题编号。

## 常见错误模式
在实际使用中，需要注意避免以下常见错误：

### 类型使用不当
- 将功能新增误用为 `chore`
- 将缺陷修复误用为 `style`
- 混淆 `refactor` 和 `feat` 的使用场景

### 描述不清晰
- 使用模糊的描述如"更新代码"
- 缺少必要的上下文信息
- 描述过长或过短

### 格式错误
- 缺少作用域括号
- 类型拼写错误
- 使用中文标点符号

### 忽略关联信息
- 未关联相关的问题编号
- 忽视破坏性变更的标记
- 遗漏必要的脚注信息

## 多代理模块协同变更
在多代理模块协同开发时，编写清晰的提交信息尤为重要：

### 协同变更原则
根据 CLAUDE.md 中的协作约定，当多个代理模块需要协同变更时：
- 明确变更的主次关系
- 在提交信息中说明协同模块
- 保持变更的一致性和原子性

### 示例场景
当 Codex 代理和 Claude 代理需要同时调整会话管理逻辑时：
```
feat(session): 统一多代理会话管理接口
为 Codex 和 Claude 代理实现统一的会话管理接口，确保行为一致性。
Closes #201
```

这种提交方式清晰地表达了变更的范围和目的，便于团队成员理解和审查。
