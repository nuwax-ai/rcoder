## 引言

本规范旨在统一项目中Rust代码的文档注释书写标准，确保所有公共接口具备清晰、完整、可生成的API文档。通过`cargo doc`工具生成的文档应能准确反映接口用途、参数说明、返回值及使用示例，提升代码可读性与可维护性。

**文档来源**  
- [lib.rs](file://crates/rcoder/src/lib.rs#L1-L16)

## 文档注释规范总则

所有公共模块、结构体、函数必须使用`///`进行文档化，以支持`cargo doc`生成高质量API文档。私有项或内部实现可根据需要使用`//`添加解释性注释。文档内容应包含：

- 功能描述
- 参数说明（使用`# 参数`）
- 返回值说明（使用`# 返回值`）
- 错误码说明（如适用）
- 使用示例（推荐）

文档注释应遵循Rust官方文档风格，使用中文书写，保持语义清晰、语法正确。

## 公共模块文档化示例

在`lib.rs`中，根模块使用`//!`进行模块级文档注释，说明库的整体功能和用途。

```rust
//! rcoder 库
//!
//! 提供 ACP 协议集成和 codex 代理管理功能
```

该注释将作为crate的顶层文档显示在`cargo doc`生成的页面中，清晰地传达库的核心职责。

## API路由函数文档块编写规范

在`router.rs`和具体处理器文件中，每个API路由函数应使用`///`进行详细文档化，并结合`utoipa`宏生成OpenAPI文档。文档块应包含请求示例和错误码说明。

### 健康检查接口文档示例

```rust
/// 健康检查端点
///
/// 检查服务的健康状态
#[utoipa::path(
    get,
    path = "/health",
    responses(
        (status = 200, description = "服务健康状态", body = HealthResponse)
    ),
    tag = "system"
)]
pub async fn health_check() -> Json<HealthResponse> {
    // ...
}
```

此文档块定义了：
- HTTP方法：GET
- 路径：`/health`
- 响应状态码与描述
- 返回体结构
- 所属标签（system）

### 聊天接口文档示例

```rust
/// 处理聊天请求 - 使用 ACP 协议集成
///
/// 发送聊天消息并获取 AI 响应。支持多媒体附件，包括文本、图像、音频和文档。
/// 如果未提供 project_id，系统会自动生成新的项目 ID 并创建相应的工作目录。
/// 如果未提供 session_id，系统会创建新的会话。
#[utoipa::path(
    post,
    path = "/chat",
    request_body(
        content = ChatRequest,
        description = "聊天请求，包含用户输入的 prompt 和可选的多媒体附件",
        content_type = "application/json"
    ),
    responses(
        (
            status = 200,
            description = "成功处理聊天请求，返回项目 ID 和会话 ID",
            body = HttpResult<ChatResponse>,
            example = json!({
                "success": true,
                "data": {
                    "project_id": "test_project",
                    "session_id": "session456",
                    "error": null
                },
                "error": null
            })
        ),
        (
            status = 400,
            description = "请求参数错误",
            body = HttpResult<String>,
            example = json!({
                "success": false,
                "data": null,
                "error": {
                    "code": "VALIDATION001",
                    "message": "Invalid request parameters"
                }
            })
        ),
        (
            status = 500,
            description = "服务器内部错误",
            body = HttpResult<String>,
            example = json!({
                "success": false,
                "data": null,
                "error": {
                    "code": "INTERNAL001",
                    "message": "Internal server error"
                }
            })
        )
    ),
    tag = "chat",
    operation_id = "handle_chat",
    summary = "发送聊天消息",
    description = "通过 ACP 协议发送聊天消息给 AI 代理，支持文本和多媒体内容"
)]
pub async fn handle_chat(
    State(state): State<Arc<AppState>>,
    Json(request): Json<ChatRequest>,
) -> Result<crate::model::HttpResult<ChatResponse>, crate::model::AppError> {
    // ...
}
```

该文档块完整描述了：
- 请求方法与路径
- 请求体结构与示例
- 各状态码的响应结构与示例
- 接口标签与操作ID
- 简要摘要与详细描述

## 复杂逻辑内部注释规范

对于内部逻辑复杂处，应使用`//`添加解释性注释，说明实现原理、关键判断条件或潜在风险点。

### 示例：Agent类型自动选择逻辑

```rust
// 根据模型提供商配置自动选择 agent 类型
// - Anthropic 协议使用 Claude Code agent
// - OpenAI 或未知协议使用 Codex agent
let agent_type = AgentType::from_model_provider(request.model_provider.as_ref());
```

### 示例：项目工作目录创建

```rust
// 创建 project_workspace 目录（如果不存在）
tokio::fs::create_dir_all(&workspace_dir).await?;

// 创建项目目录
let project_dir = workspace_dir.join(project_id);
tokio::fs::create_dir_all(&project_dir).await?;
```

此类注释有助于后续维护者快速理解代码意图，降低理解成本。

## 使用示例代码块规范

鼓励为公共接口提供`# Examples`代码块，展示典型使用场景。示例应简洁、完整、可运行。

### 示例：HttpResult使用示例

虽然未直接在源码中体现，但推荐在公共API中添加如下形式的使用示例：

```rust
/// 创建成功的响应结果
///
/// # Examples
///
/// ```
/// use rcoder::model::HttpResult;
///
/// let result = HttpResult::success("OK");
/// assert!(result.success);
/// ```
///
/// 创建错误的响应结果
///
/// # Examples
///
/// ```
/// use rcoder::model::HttpResult;
///
/// let result = HttpResult::error("400", "Bad Request");
/// assert!(!result.success);
/// ```
```

此类示例显著提升API的可用性，帮助使用者快速掌握接口用法。

## 总结

本文档确立了项目中Rust代码的文档注释标准：
1. 所有公共项使用`///`进行文档化
2. API路由函数需包含请求示例和错误码说明
3. 复杂逻辑处使用`//`添加解释性注释
4. 鼓励为公共接口提供`# Examples`代码块
5. 利用`utoipa`等工具生成OpenAPI文档

遵循此规范可确保代码具备良好的自描述性，便于团队协作与长期维护。
