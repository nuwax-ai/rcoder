## 项目结构
RCoder项目采用Rust多包工作区结构，核心组件包括：
- `crates/rcoder`：主服务，基于Axum框架
- `crates/pingora-proxy`：高性能反向代理服务
- 多个AI代理模块（Claude、Codex等）
- Shell测试脚本位于根目录，用于自动化验证代理功能

测试脚本通过启动Python HTTP服务器模拟后端，调用本地构建的`rcoder`二进制文件启动代理服务，并使用`curl`进行端到端验证。

## 核心测试脚本分析

### test_proxy.sh：代理连通性测试
该脚本实现完整的端到端代理功能验证流程：
1. 启动两个Python HTTP服务器作为后端（端口3000和3001）
2. 启动RCoder代理服务，监听8080端口，指向默认后端3000
3. 通过`curl`和`grep`组合测试多种代理路径：
   - 查询参数方式：`/proxy?port=3000`
   - 路径参数方式：`/proxy/3000/`
   - 默认端口代理
4. 验证响应内容是否包含"Directory listing for"标识
5. 测试完成后自动清理进程

脚本使用`set -e`确保任何命令失败即终止，并通过`sleep`等待服务启动。

**Section sources**
- [test_proxy.sh](file://test_proxy.sh#L1-L90)

### test_proxy_api.sh：API接口测试
该脚本专注于验证代理API的多种使用场景：
- 路径参数代理：`/proxy/8080/api/users`
- 仅端口代理：`/proxy/8080`
- 带查询参数的GET请求
- 向后兼容的查询参数方式：`/proxy?port=8080&path=/api/users`
- POST请求带JSON体

使用`curl -s -w "\n状态码: %{http_code}\n"`同时输出响应体和HTTP状态码，便于快速判断测试结果。

**Section sources**

### test_query_params.sh：查询参数传递测试
专门验证查询参数的完整传递功能：
- 普通查询参数：`param1=value1&param2=value2`
- POST请求中的查询参数
- 特殊字符编码：`q=rust%20programming`

使用`curl -v`输出详细请求信息，并通过`grep`过滤关键字段（GET、Host、HTTP状态码等），确保查询参数被正确转发。

## 代理机制与API测试实践

### 代理模式对比
RCoder支持两种代理模式：
1. **路径参数模式**（推荐）：`/proxy/{port}/{path}`
2. **查询参数模式**（向后兼容）：`/proxy?port=8080&path=/api/users`

```mermaid

A[客户端请求] --> B{请求格式}
B --> |路径格式| C[/proxy/8080/api/users]
B --> |查询参数| D[/proxy?port=8080&path=/api/users]
C --> E[直接路由到端口8080]
D --> F[解析port参数]
F --> E
E --> G[后端服务响应]
G --> H[客户端]
```

### 认证头处理
测试脚本通过`-H "Accept: application/json"`等参数验证代理服务正确传递HTTP头。生产环境中可类似添加认证头：
```bash
curl -H "Authorization: Bearer token123" http://localhost:8080/proxy/8080/api/data
```

### JSON响应结构校验
虽然当前脚本主要验证状态码和基本响应，但可通过`jq`扩展实现JSON结构校验：
```bash
curl -s http://localhost:8080/proxy/8080/api/users | jq -e '.data[]?.id, .meta.total'
```

## 流式响应（SSE）处理
尽管当前测试脚本未直接涉及SSE，但项目中存在SSE流式响应处理逻辑。对于SSE端点的测试，应使用：
```bash
curl --no-buffer --raw http://localhost:8080/sse-endpoint \
  | grep -m 1 "data:" \
  | sed 's/data: //' \
  | jq -e '.expected_field'
```
`--no-buffer`确保实时接收事件，`--raw`保留原始格式，便于`grep`和`sed`提取关键字段进行断言。

## 稳定性最佳实践

### 超时控制
在生产测试中应添加超时：
```bash
curl --max-time 30 --fail http://localhost:8080/proxy/8080/health
```

### 重试机制
实现简单重试逻辑：
```bash
retry() {
  local n=0
  local max=3
  while [[ $n -lt $max ]]; do
    if "$@"; then
      return 0
    else
      ((n++))
      sleep 2
    fi
  done
  return 1
}
retry curl -f http://localhost:8080/proxy/8080/ready
```

### 日志捕获
如`test_proxy.sh`所示，将代理服务日志重定向到文件：
```bash
./rcoder > logs/proxy_test.log 2>&1 &
```
测试完成后使用`tail`或`grep`分析关键日志。

## 结论
RCoder的Shell测试脚本提供了一套完整的端到端测试实践，有效验证了反向代理的核心功能。通过`test_proxy.sh`、`test_proxy_api.sh`和`test_query_params.sh`三个脚本，覆盖了服务启动、多种代理模式、查询参数传递等关键场景。建议在后续开发中：
