## 引言

rcoder的工具插件架构旨在为AI代理提供安全、可扩展且结构化的功能调用能力。该系统通过Rust实现高性能工具，并利用JSON Schema将函数接口暴露给语言模型理解与调用。工具涵盖文件系统操作、网络搜索、时间查询等常见任务，同时通过权限控制机制确保敏感操作的安全性。本文档系统性地文档化该架构的设计原理、执行流程与扩展方法。

## 工具架构概览

rcoder的工具系统位于`crates/agent2/src/tools/`目录下，采用模块化设计，每个工具独立实现并统一注册。所有工具均实现`AgentTool` trait，确保接口一致性。工具通过`tool_schema`模块生成符合语言模型理解的JSON Schema，并通过事件流机制与代理进行异步通信。

## 核心工具设计模式

### 文件读取工具 (read_file_tool)

`ReadFileTool`用于读取项目中的文件内容，支持按行范围读取。对于大文件，自动返回结构化大纲而非完整内容，避免上下文溢出。

```mermaid
sequenceDiagram
participant AI as AI代理
participant Tool as ReadFileTool
participant Project as Project
participant Buffer as Buffer
participant FS as 文件系统
AI->>Tool : 调用read_file(path="src/main.rs", start_line=10)
Tool->>Project : find_project_path("src/main.rs")
alt 路径有效
Project-->>Tool : 返回ProjectPath
Tool->>Project : absolute_path()
Tool->>WorktreeSettings : 检查路径是否被排除或私有
alt 路径允许访问
Tool->>Project : open_buffer()
Tool->>Buffer : text_for_range(start..end)
Buffer-->>Tool : 文件内容
Tool->>AI : 返回文本结果
else 路径被排除
Tool-->>AI : 返回权限错误
end
else 路径无效
Tool-->>AI : 返回"路径未找到"错误
end
```

## 工具权限控制机制

### 权限检查流程

文件操作类工具在执行前会进行多层次权限检查，防止访问敏感或外部文件。

```mermaid
flowchart TD
Start([开始]) --> GetPath["获取输入路径"]
GetPath --> Resolve["解析为ProjectPath"]
Resolve --> Valid{"路径有效?"}
Valid --> |否| Error1["返回路径未找到错误"]
Valid --> |是| CheckGlobal["检查全局设置"]
CheckGlobal --> ExcludedGlobal{"路径在file_scan_exclusions中?"}
ExcludedGlobal --> |是| Error2["返回全局排除错误"]
ExcludedGlobal --> |否| CheckPrivateGlobal{"路径在private_files中?"}
CheckPrivateGlobal --> |是| Error3["返回全局私有错误"]
CheckPrivateGlobal --> |否| CheckWorktree["检查工作树设置"]
CheckWorktree --> ExcludedWorktree{"路径在工作树exclusions中?"}
ExcludedWorktree --> |是| Error4["返回工作树排除错误"]
ExcludedWorktree --> |否| CheckPrivateWorktree{"路径在工作树private_files中?"}
CheckPrivateWorktree --> |是| Error5["返回工作树私有错误"]
CheckPrivateWorktree --> |否| Allow["允许访问"]
Allow --> End([结束])
Error1 --> End
Error2 --> End
Error3 --> End
Error4 --> End
Error5 --> End
```

## 自定义工具开发教程

### 开发步骤

1. **创建工具模块**：在`crates/agent2/src/tools/`下创建新文件（如`my_tool.rs`）
2. **定义输入输出结构体**：使用`#[derive(Serialize, Deserialize, JsonSchema)]`
3. **实现AgentTool trait**
4. **在tools.rs中注册**
5. **编写测试**

### 最佳实践

#### 参数验证
- 使用`serde`的`default`属性提供默认值
- 在`run`方法中尽早验证输入有效性
- 返回清晰的错误信息

#### 错误处理
- 使用`anyhow::Result`进行错误传播
- 对用户友好的错误消息
- 记录调试信息

#### 结果序列化
- 实现`Into<LanguageModelToolResultContent>`用于输出
- 对于复杂输出，考虑使用`serde(transparent)`包装
- 确保输出可被语言模型理解

```mermaid
flowchart TD
A[创建my_tool.rs] --> B[定义MyToolInput]
B --> C[定义MyTool]
C --> D[实现AgentTool]
D --> E[在tools.rs中添加mod和use]
E --> F[实现default_tool_names]
F --> G[编写单元测试]
G --> H[完成]
```

## 结论

rcoder的工具生态系统通过标准化的`AgentTool`接口、灵活的`tool_schema`机制和严格的权限控制，构建了一个安全、可扩展的AI代理功能调用框架。`acp_tools`进一步提供了通用的调试和监控能力。开发者可以遵循清晰的模式快速创建自定义工具，为AI代理赋予新的能力。该架构平衡了功能性、安全性和可维护性，为智能编程助手的发展提供了坚实基础。
