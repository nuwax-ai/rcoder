# 修复 Agent 验证命令导致阻塞的问题

## 问题分析

### 当前问题
日志显示：`"2025-12-08T07:40:55.527750Z DEBUG 运行验证命令: claude-code-acp ["--version"]"`

**根本原因**：
- `claude-code-acp --version` 命令会启动一个持续运行的进程，不会退出
- 这导致验证流程阻塞，无法继续

### 代码位置
1. **manager.rs:138-158** - `ensure_installed()` 方法
   - 调用 `validate_installation()` 检查命令是否可用
   
2. **manager.rs:185-209** - `validate_installation()` 方法
   - 当前逻辑：
     - 如果配置了 `validate_command`，使用自定义验证命令
     - 否则使用 `which::which()` 检查命令是否在 PATH 中
     - **如果 which 失败，会回退到 `try_version_check()`**
   
3. **manager.rs:239-264** - `try_version_check()` 方法
   - **问题所在**：尝试运行 `command --version` 和 `command --help`
   - 对于 `claude-code-acp`，这会导致程序挂起

### 现有的正确逻辑
- `is_command_available()` 使用 `which::which()` 检查命令
- 这是正确且高效的方式，不会执行命令

## 解决方案

### 方案选择：移除 `--version` 回退逻辑

**理由**：
1. `which::which()` 已经足够可靠，是标准的命令检测方式
2. 运行 `--version` 可能导致程序启动（如 claude-code-acp）
3. 运行 `--version` 可能有副作用（日志、网络请求等）
4. 用户可以通过 `validate_command` 配置自定义验证

### 实施步骤

#### 1. 修改 `manager.rs::validate_installation()`
**文件**: `crates/agent_config/src/installer/manager.rs:185-209`

```rust
// 当前逻辑
pub async fn validate_installation(
    &self,
    config: &InstallationConfig,
    command: &str,
) -> Result<bool, InstallationError> {
    // ... 检查 validate_command ...
    
    // 使用 which 检查
    if self.is_command_available(command).await {
        return Ok(true);
    }

    // ❌ 问题：回退到 --version 检查
    let result = self.try_version_check(command).await;
    Ok(result)
}
```

**修改为**：
```rust
pub async fn validate_installation(
    &self,
    config: &InstallationConfig,
    command: &str,
) -> Result<bool, InstallationError> {
    // ... 检查 validate_command ...
    
    // ✅ 只使用 which 检查，不再回退
    Ok(self.is_command_available(command).await)
}
```

#### 2. 删除 `try_version_check()` 方法
**文件**: `crates/agent_config/src/installer/manager.rs:239-264`

- 这个方法不再使用，可以安全删除
- 减少代码复杂度
- 避免误用

#### 3. 更新注释和文档
- 更新 `validate_installation()` 的注释，说明只使用 which 检查
- 如果需要自定义验证逻辑，使用 `validate_command` 配置

## 影响分析

### 正面影响
1. ✅ 解决 claude-code-acp 验证阻塞问题
2. ✅ 验证速度更快（不执行命令）
3. ✅ 避免命令执行的副作用
4. ✅ 代码更简洁

### 潜在风险
1. ⚠️ 某些命令可能在 PATH 中但损坏/不可执行
   - **缓解方案**：实际运行时会自然失败并报错
   - **缓解方案**：用户可以配置 `validate_command` 进行深度检查

2. ⚠️ 向后兼容性
   - **影响评估**：低
   - **原因**：这是内部实现细节，外部 API 不变
   - **原因**：which 检查已经是主要逻辑，--version 只是回退

### 测试计划
1. 单元测试：`test_command_availability()` 已存在
2. 集成测试：启动 claude-code-acp agent
3. 手动测试：
   - 验证 claude-code-acp 不再阻塞
   - 验证不存在的命令返回 false
   - 验证存在的命令返回 true

## 替代方案（不推荐）

### 方案 B：添加超时机制
```rust
async fn try_version_check(&self, command: &str) -> bool {
    let result = tokio::time::timeout(
        Duration::from_secs(2),
        tokio::process::Command::new(command).arg("--version").output()
    ).await;
    
    matches!(result, Ok(Ok(output)) if output.status.success())
}
```

**不推荐原因**：
- 增加复杂度
- 仍然会启动进程（副作用）
- 超时时间难以确定
- which 检查已经足够

## 总结

**推荐方案**：移除 `try_version_check()` 回退逻辑

**关键改动**：
1. `validate_installation()` - 移除 `try_version_check()` 调用
2. 删除 `try_version_check()` 方法
3. 更新相关注释

**预期结果**：
- claude-code-acp 验证不再阻塞
- 验证速度提升
- 代码更简洁可维护
