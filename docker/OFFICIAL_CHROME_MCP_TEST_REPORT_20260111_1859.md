# 官方 Chrome DevTools MCP 测试报告 (带缓存)

**时间**: 2026-01-11 18:59
**测试目标**: 验证在预安装 MCP 工具 (`npm install -g`) 并配置持久化缓存 (`npm_config_cache=/root/.npm`) 后，使用官方 `npx chrome-devtools-mcp` 命令的性能表现。

## 测试环境
- **镜像**: `rcoder-agent-runner` (基于 `rcoder-agent-base` 重建)
- **MCP 配置**:
  ```json
  "chrome-devtools": {
    "command": "npx",
    "args": ["-y", "chrome-devtools-mcp@latest"]
  }
  ```
- **并发**: 15 线程 x 3 轮
- **间隔**: 轮次间间隔 5 秒

## 测试结果

| 轮次 | 平均耗时 | 最长耗时 | 成功率 | 备注 |
|---|---|---|---|---|
| R1 | ~3.8s | 4.8s | 100% | 非常快，证明 npx 使用了本地缓存，未下载 |
| R2 | ~6.5s | 7.6s | 100% | 依然流畅，轻微性能下降 |
| R3 | ~51s | 52.1s | 100% | 性能明显下降，但**无超时** (未触及 100s 阈值) |

## 结论
1.  **缓存方案有效**：R1 的极速响应证明 `npx` 没有执行网络下载，而是直接使用了预安装/缓存的包。
2.  **解决死锁/阻塞**：虽然 R3 耗时增加到 50s，但所有请求最终都成功返回，没有出现之前的 100s 超时或彻底卡死现象。
3.  **资源开销**：Official Chrome MCP 每个 Agent 都会启动独立的 `node` 和 `chromium` 进程，资源开销较大，导致 R3 性能下降。相比之下，`mcp-proxy` (共享 Chromium) 在 R3 表现更好 (~8s)，但架构更复杂。
4.  **可行性**：当前方案（官方插件+缓存优化）是**可行的**，能满足基本稳定运行需求。
