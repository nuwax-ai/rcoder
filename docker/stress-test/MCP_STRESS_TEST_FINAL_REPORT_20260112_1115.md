## 资源分析

### 容器累积问题

| 轮次 | 容器数量 (累计) | 内存压力 | 状态 |
|------|----------------|----------|------|
| R1 | 15 | 低 | ✅ |
| R2 | 30 | 中 | ⚠️ |
| R3 | 45 | 高 | ⛔ |

### 根因分析

1. **容器未及时清理**: 每轮创建 15 个容器，累积到 45 个时资源耗尽
2. **MCP 进程占用**: 每个容器有 10 个 MCP 进程 → 45 × 10 = 450 个进程
3. **内存不足**: Docker 宿主机内存被消耗殆尽
4. **连接拒绝**: HTTP=000 表示 curl 无法建立连接（服务无响应）

## 建议

### 🔴 生产环境

1. **MCP 工具数量**: 建议 ≤ 3 个
2. **并发限制**: 建议 ≤ 10 并发
3. **容器清理**: 每轮测试后清理，或设置 TTL 自动清理

### 🟡 测试环境

1. **轮次限制**: 连续测试不超过 2 轮
2. **轮次间清理**: 使用 `./docker/scripts/cleanup.sh`
3. **资源监控**: 监控 Docker 宿主机内存

### 🟢 优化方向

1. **MCP 复用**: 使用 mcp-proxy 共享 MCP 实例
2. **预装包**: 将常用 npx 包预装到 Dockerfile
3. **容器 TTL**: 缩短容器自动清理时间
