# Centralized Shared MCP Strategy 验证报告

> **测试时间**: 2026-01-12 14:35 - 15:00
> **测试策略**: 环境变量替换 (`${MCP_PROXY_URL:-...}`) + 动态动态切换
> **启动类型**: 经典模式 vs 共享模式 (含 3 副本扩展测试)

---

## 1. 核心结论

| 模式 | 配置 | 内存占用 | 速度 | 45并发稳定性 | 隔离性 |
|------|------|----------|------|--------------|--------|
| **经典模式** | 本地 Proxy | 高 (每 Agent +200MB) | 快 (8s) | ⛔ 崩溃 (宿主机 OOM) | ✅ 强 (独立实例) |
| **共享模式** | 单实例 Proxy | 低 (总量固定) | 慢 (37s) | ⛔ 超时 (单点瓶颈) | ⚠️ 弱 (有串台风险) |
| **共享扩展** | 3 副本 Proxy | 中 (总量 x3) | 中 (37s) | ⛔ 超时 (30并发后瓶颈) | ⚠️ 弱 (负载均衡非隔离) |

---

## 2. 深度分析：如何支持 45+ 并发？

用户提问：*“如何扩展一下以便支持45个？”*

我们进行了 **3 副本 (Replicas=3)** 的实测：
- **配置**: 在 `docker-compose.yml` 中设置 `deploy: replicas: 3`。
- **结果**:
    - **R1 (15并发)**: ✅ 通过 (38s)。负载均衡有效。
    - **R2 (30并发)**: ⛔ 失败 (超时)。
    - **原因**: 宿主机在运行 3 个 Chrome 实例 + 30 个 Agent 容器时，再次遭遇资源/网络瓶颈。
- **建议**:
    1.  **分离部署**: 将 `mcp-proxy-host` 部署在独立机器上，而非与 Agent 跑在同一台开发机。
    2.  **资源配额**: 此时瓶颈已从“软件配置”转移到了“硬件资源”。支持 45 个并发 Chrome Session 需要显著的 CPU/内存资源。

---

## 3. 风险提示：消息串台 (Crosstalk)

用户提问：*“会不会存在消息串的问题？”*

**答案：是，存在风险。**

- **无状态工具 (如 `trends-hub` HTTP)**:
    - **风险低**。如果是单纯的数据查询，共享 Proxy 只是作为一个 HTTP 网关，各 Agent 请求相互独立。
- **有状态工具 (如 `chrome-devtools` 浏览器)**:
    - **风险极高**。
    - 所有连接到同一个 `mcp-proxy-host` (或同一个副本) 的 Agent，实际上是在操作**同一个 Chrome 实例**。
    - **后果**: Agent A 打开的标签页，Agent B 列出标签页时能看到。Agent A 的 Cookie 可能与 Agent B 混用。
- **结论**:
    - 共享模式适用于**只读/无状态**任务（如搜索、查天气）。
    - **严禁**用于操作敏感账户或需要独立浏览器上下文的任务，除非 MCP 服务端本身支持 Session 隔离（目前 `mcp-server-chrome-devtools` 主要是单实例设计）。

---

## 4. 最终建议

1.  **默认开发**: 使用 **Classic Mode** (默认)。最简单，隔离性最好，适合 <20 并发调试。
2.  **基准压测**: 使用 **Shared Mode** (单实例)。仅用于测试 Agent 逻辑本身的吞吐量，忽略浏览器隔离性。
3.  **高并发生产**: **不建议使用共享 Chrome**。应采用 headless 浏览器集群 (如 Browserless.io) 或为每个 Agent Pod 注入 Sidecar (即回归 Classic 模式但优化资源)。

## 附录：动态切换配置

**无需重新打包**，修改 `docker/config.yml` 即可：

```yaml
# 开启共享模式 (节省内存)
ENABLE_MCP_PROXY: "false"
```

```yaml
# 回归经典模式 (隔离性好)
ENABLE_MCP_PROXY: "true"
```

重启生效:
```bash
docker-compose -f docker/docker-compose.yml down
docker-compose -f docker/docker-compose.yml up -d
```
