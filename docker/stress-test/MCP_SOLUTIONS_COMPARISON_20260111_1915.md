# MCP 方案对比报告: Chrome DevTools
**时间**: 2026-01-11 19:15

基于 15 并发 x 3 轮的压力测试，对比了两种 Chrome DevTools MCP 配置方案。

## 1. 方案对比

| 方案 | 配置方式 | 机制 | R1 耗时 | R3 耗时 | 稳定性 | 复现脚本 | 结论 |
|---|---|---|---|---|---|---|---|
| **A. 官方 npx (带缓存)** | `command: npx` <br> `args: [-y, chrome-devtools-mcp@latest]` | 每个 Agent 启动独立 Chromium 进程 (利用预安装缓存) | **3.8s** | ~50s | ✅ **稳定** (无超时) | `stress_test_official_chrome.sh` | **推荐** |
| **B. 本地 Proxy (mcp-proxy)** | `command: mcp-proxy` <br> `args: [convert, http://127.0.0.1:18099]` | Agent 连接容器内常驻的 mcp-proxy 服务 | **103s (超时)** | 崩溃 | ❌ **严重阻塞** | `stress_test_proxy_chrome.sh` | **不可用** |

> **备注**: 方案 B 测试时使用了用户提供的 `chrome-tools` key 配置，除了 key 名称外，其他参数与之前测试完全一致。

## 2. 详细测试数据

### 方案 A (npx) [推荐]
- **R1**: 平均 3.8s (最长 4.8s) - **极速**，证明缓存生效。
- **R3**: 平均 50s - 资源负载高导致变慢，但**所有请求成功返回**。

### 方案 B (mcp-proxy) [失败]
- **R1**: 平均 103s (超时) - 大量请求在 100s+ 后才返回或超时。
- **R2**: 触发服务崩溃或拒绝连接 (HTTP 000)。
  ```
  ❌ R2-12: HTTP=000 .893s
  ❌ R3-1: HTTP=000 .011s
  ```

## 3. 结果分析
- **npx 方案**：虽然高并发下资源开销大（R3 变慢），但胜在**隔离性好**。通过 Dockerfile 预安装和 `npm` 缓存持久化配置，消除了运行时的下载延迟，实现了 R1 的秒级响应。
- **mcp-proxy 方案**：虽然理论上共享连接更高效，但在高并发下出现了严重的**死锁或资源争抢**，导致请求直接挂起直至超时。这可能是 `mcp-proxy` 服务端处理并发连接的瓶颈导致的。

## 4. 最终配置
已将配置恢复为经过验证的 **方案 A (npx)**，以确保系统的稳定性和可用性。
