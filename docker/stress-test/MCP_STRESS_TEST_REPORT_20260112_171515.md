## 🔧 脚本使用指南 (Script Usage)

本测试依赖以下三个核心脚本：

### 1. 启动压测 (Start Stress Test)
会生成全新的 `BATCH_ID` 并开始压测。
```bash
# 用法: ./stress_test_10mcp_remote.sh [并发数] [轮次]
./docker/scripts/stress_test_10mcp_remote.sh 15 3
```

### 2. 系统诊断 (Diagnose)
**推荐！** 在压测进行中或结束后，使用此脚本分析特定批次的健康状况。
```bash
# 用法: ./diagnose_remote_system.sh [BATCH_ID]
./docker/scripts/diagnose_remote_system.sh <BATCH_ID>
```

### 3. 远程服务重启 (Reset)
如果发现超时严重且无法自行恢复，可以使用此脚本重启远程服务（会清理所有容器）。
```bash
./docker/scripts/restart_remote_service.sh
```

## 💡 最佳实践建议

1. **关于容器复用**:
   * **推荐**: 尽量开启容器复用（响应时间 3-6s）。
   * **若必须不复用**: 请确保单次请求处理完后，容器能被快速清理。残留容器是导致性能雪崩的元凶。

2. **关于并发控制**:
   * 在不复用容器模式下，**15 并发** 是一个临界点。如果服务器残留容器 > 30 个，新请求可能会触发 100s 超时。建议控制在 10 并发以内以保稳定。

3. **关于 MCP 数量**:
   * 10 个 MCP 的冷启动时间约为 **12-18s** (在资源充足时)。如果在业务上无法接受此延迟，必须减少默认启动的 MCP 数量。

### 容器状态分析

- **累积效应**: 每个批次启动 45 个容器（15 × 3），5 个批次理论上最多累积 225 个容器
- **清理机制**: R3 性能的波动说明容器清理机制在运行，但清理速度跟不上累积速度
- **资源竞争**: Batch 3 开始，系统资源严重不足，导致所有请求响应时间 > 100s

### 关键发现

1. **性能雪崩点**: Batch 3 是性能雪崩的临界点，R1 就已经超过 100s
2. **容器清理延迟**: 容器清理机制存在，但清理速度不足以应对 15 并发 × 3 轮的累积
3. **恢复能力**: R3 的性能恢复说明系统仍有恢复能力，但需要更长的清理时间
4. **超时保护有效**: 虽然响应时间超过 100s，但超时保护机制成功阻止了无限等待

### 结论与建议

#### 核心结论
1. **单批次测试可行**: Batch 1 的 R1 表现正常（15s），说明单批次 15 并发 × 3 轮在干净环境下可行
2. **多批次累积不可行**: 连续 5 个批次导致系统资源严重不足，Batch 3 开始全面超时
3. **容器清理是关键**: 容器清理速度是系统稳定性的关键因素

#### 改进建议

1. **降低并发数**: 
   - 从 15 降至 10 或更少
   - 或增加批次之间的等待时间（建议 30-60 秒）

2. **优化容器清理**:
   - 实现更激进的容器清理策略
   - 在请求完成后立即清理容器，而不是等待超时

3. **启用容器复用**:
   - 使用相同的 user_id 以复用容器
   - 可大幅提升性能（响应时间从 15-100s 降至 3-6s）

4. **监控与告警**:
   - 监控运行中容器数量
   - 当容器数量 > 30 时发出告警
   - 自动触发容器清理或拒绝新请求

5. **批次间隔优化**:
   - 在批次之间增加更长的等待时间（建议 60 秒）
   - 允许容器清理机制有足够时间工作
