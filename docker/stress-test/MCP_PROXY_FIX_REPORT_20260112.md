# MCP Proxy 问题排查报告: 资源优化方案
> **测试时间**: 2026-01-12 12:55
> **结论**: 所谓的"死锁/连接复用"问题实为**资源耗尽 (OOM)**。

## 1. 问题根因
- **现象**: 压力测试在 R3 (45 容器) 时全部失败 (HTTP 000, 连接拒绝)。
- **分析**: 默认情况下，每个 Agent 容器都会启动一个 `mcp-proxy` 服务，该服务会拉起一个 **Chromium 浏览器实例** (用于 support chrome-devtools)。
- **资源开销**: 45 个容器 x 1 个 Chromium 实例 = **巨大的内存开销**。
- **关键点**: 实际上，本次压力测试使用的是 `trends-hub` 或 `10-Mcp` 工具，**完全没有使用** 这个共享的 Chromium 代理。这些资源被白白浪费，最终导致宿主机内存耗尽，服务崩溃。

## 2. 误区澄清
- 最初怀疑 `mcp-proxy` 存在连接复用锁问题。
- 经查证，该 Rust 版本 `mcp-proxy` 不支持 `--stateless` 参数，且 Agent 使用的是独立的 `convert` 进程，不存在跨容器的锁复用。
- 问题不在于锁，而在于**过重的默认配置**。

## 3. 解决方案
在 `start-up.sh` 中增加了**环境开关**，允许按需关闭共享代理服务。

- **开关**: `ENABLE_MCP_PROXY` (默认 `true`)
- **优化**: 当设置为 `false` 时，不启动共享 `mcp-proxy` 和 Chromium，每容器节省约 200MB+ 内存。

## 4. 验证结果 (禁用代理后)

| 轮次 | 容器数 | 结果 | 耗时 |
|------|--------|------|------|
| R1 | 15 | (因重启服务未就绪略过) | - |
| **R2** | 30 | ✅ **100% 成功** | 36s |
| **R3** | 45 | ✅ **100% 成功** | 40s |

> 对比: 优化前 R3 100% 失败。优化后系统运行平稳。

## 5. 建议
1. **日常开发**: 保持默认 (开启)，以便支持 Chrome 自动化 Agent。
2. **压力测试**: 在 `config.yml` 中设置 `ENABLE_MCP_PROXY: "false"`，以避免无意义的资源消耗。
