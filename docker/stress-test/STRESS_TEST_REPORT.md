## 代码级问题定位

### 阻塞点代码 (`claude_code.rs` L614-628)

```rust
// 添加 100 秒超时保护（MCP 工具较多时启动较慢）
let resp = tokio::time::timeout(
    tokio::time::Duration::from_secs(100),
    client_conn.new_session(new_session_request), // ⬅️ 阻塞点
)
.map_err(|_| {
    error!("🔴 [ACP] new_session 超时 (100s)! MCP 服务器: {:?}", mcp_servers);
    anyhow!("ACP 会话创建超时 - Agent 子进程可能卡在 MCP 服务器连接")
});
```

### 问题链路
```
请求 → new_session() → MCP 服务器初始化(可能卡住) → 100s 超时
                         ↑
                   问题根源：MCP 工具初始化阻塞
```

### 为什么本次测试未复现
| 因素 | 生产环境 | 本次测试 |
|------|----------|----------|
| MCP 工具数 | 10+ 个 | 0 个 |
| context_servers | 完整配置 | 无 |
| 外部服务依赖 | 网络调用 | 无 |

### 🎯 根因推断
**MCP 服务器初始化** 在高并发时可能：
1. 同时启动多个 `mcp-proxy` 进程
2. 外部服务连接超时
3. 资源竞争导致初始化变慢


### 验证步骤
1. 在测试环境复现问题（多开 Agent）
2. 搜索关键字：`开始创建 ACP 会话`
3. **如果多个 Agent 都有这个日志** → 线程独立，问题在 claude-code-acp 黑盒
4. **如果只有前几个有日志** → 线程被阻塞，问题在 LocalSet 串行

### 黑盒边界
```
Rust 代码 (可控)          TS 黑盒 (不可控)
     ↓                        ↓
new_session() ─────────→ claude-code-acp
     │
     └── 日志：🔵 开始创建
```
