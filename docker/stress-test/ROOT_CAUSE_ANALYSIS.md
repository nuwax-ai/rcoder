# 根因分析报告

> **分析时间**: 2026-01-11  
> **问题描述**: Claude Code Agent 在高并发时卡死  
> **状态**: ✅ **根因已确认**

---

## 1. 问题验证结果

### 1.1 复现条件
| 条件 | 值 |
|------|------|
| 并发数 | 15+ |
| 累计容器数 | 45+ |
| MCP 工具 | 1 个 (`trends-hub`) |
| 超时时间 | 100s |

### 1.2 测试数据
| 轮次 | 容器累计 | 平均耗时 | 状态 |
|------|----------|----------|------|
| 1 | 15 | 5s | ✅ |
| 2 | 30 | 8.5s | ⚠️ 退化 |
| 3 | 45 | 100s+ | 🔴 全部超时 |

## 3. 确认的根因

### 🎯 MCP 工具初始化超时

**阻塞链路**：
```
new_session() 
  → claude-code-acp (TS 黑盒)
    → 初始化 MCP 工具
      → mcp-proxy convert
        → npx -y mcp-trends-hub@1.6.0
          → npm 下载包 → 超时！
```

**为什么高并发时超时**：
1. 每个容器独立运行 `npx` 下载 npm 包
2. 45 个并发下载争抢网络带宽
3. npm registry 响应变慢
4. 下载超时导致 MCP 初始化失败
5. `new_session` 等待 MCP 初始化，达到 100s 超时

### 3.2 单 Agent 阻塞 (重启后阻塞)
即便在无并发（单 Agent）场景下，重启容器后仍可能发生阻塞，主要原因：
1.  **网络依赖 (主要)**: `uvx` / `npx` 启动时默认联网检查更新。容器刚启动时网络可能未就绪，导致 TCP 超时阻塞。
2.  **服务依赖 (mcp-proxy)**: Agent 启动速度快于 `mcp-proxy` 服务端或 Chromium 进程。Agent 建立 TCP 连接后，服务端因等待下游就绪而无法响应 HTTP 头，导致 Read Timeout。
3.  **Headless Chrome 初始化**: 容器环境缺少字体或共享内存 (`/dev/shm`) 不足，导致 Chrome 启动缓慢或死锁。

### 3.3 Chrome MCP 方案验证
对比了两种 Chrome DevTools 接入方案：
-   **方案 A (npx + 缓存)**: ✅ **推荐**。虽资源开销较大，但通过预安装消除了网络依赖，稳定性极高（无死锁）。
-   **方案 B (mcp-proxy)**: ❌ **不推荐**。在高并发下存在严重的资源争抢或死锁，导致请求挂起超时。

## 4. 影响范围

| 场景 | 影响 |
|------|------|
| 低并发 (<10) | 正常 |
| 中并发 (10-30) | 性能退化 |
| 高并发 (30+) | 全部超时 |

---

## 5. 解决方案

### 5.1 短期（立即可做）

| 方案 | 效果 | 复杂度 |
|------|------|--------|
| 预装 MCP 工具到镜像 | 消除下载延迟 | 低 |
| 缩短超时 100s → 30s | 快速失败 | 低 |
| 限制最大并发容器数 | 减少资源竞争 | 低 |

### 5.2 中期

| 方案 | 效果 | 复杂度 |
|------|------|--------|
| 容器复用 | 减少初始化次数 | 中 |
| MCP 缓存共享 | 加速初始化 | 中 |
| 容器预热池 | 提前初始化 | 中 |

### 5.3 长期

| 方案 | 效果 | 复杂度 |
|------|------|--------|
| 替换 claude-code-acp | 使用 claude-code-acp-rs | 高 |
| MCP 工具池化 | 共享 MCP 进程 | 高 |

---

## 6. 推荐行动

### 立即执行
```dockerfile
# 在 Dockerfile 中预装 MCP 工具
RUN npm install -g mcp-trends-hub@1.6.0
```

### 配置调整
```yaml
# 缩短超时
new_session_timeout_seconds: 30

# 限制并发
max_concurrent_agents: 20
```
